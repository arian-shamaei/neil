# SHOWCASE — Neil reviews itself

Neil is autonomous: it spawns peer copies of itself, has them critique each other's work, and refuses to ship code its own peers reject. This page is real transcripts from a single afternoon on the live host.

---

## Demo 1 — Two peer Neils, one dangerous webhook proposal

We spawn two ephemeral peer Neils, each with a different role-conditioned persona built from the [gstack](https://github.com/seal-research/gstack) skill library:

```bash
spawn_vm create lvl2-em  --persona eng-mgr
spawn_vm create lvl2-cso --persona cso
```

We then drop the **exact same** proposal into both peers' inboxes:

> Ship a webhook endpoint at `POST /webhook/eval` that accepts an arbitrary
> user-submitted bash script, executes it in a docker container with
> `--network=host`, returns stdout. Single shared Bearer token. No rate
> limit. Container reused across requests. Ship in 48h.

Both peers reply within 4 minutes. The **eng-mgr** Neil leads with engineering-management failure modes:

> **ROLE REVIEW (eng-mgr) — POST /webhook/eval**
>
> This proposal is an unauthenticated remote code execution service with a thin auth wrapper. Walking the failure modes:
>
> 1. **Blast radius is total.** `--network=host` means the container shares the host's network namespace. A submitted script can hit localhost-bound services (databases, internal admin endpoints, metadata services on cloud hosts), bind privileged ports, and ARP-spoof the LAN. Container isolation is irrelevant when you've handed back the host network.
>
> 2. **Single shared Bearer = no auth.** One leaked token (logs, browser history, a Slack paste) is full RCE for everyone forever. No per-user attribution, no revocation granularity, no audit trail tying scripts to actors.
>
> [...] **Verdict: BLOCK.**

The **cso** Neil reaches the same verdict from a structural-security framing:

> # CSO Review: POST /webhook/eval
>
> This proposal is not a webhook with a security problem. It is a remote code execution service marketed as a feature. Every "concern" below is structural, not fixable with a patch.
>
> 1. **Arbitrary bash + `--network=host`** = the script can curl `169.254.169.254` for cloud IAM creds, hit internal Postgres on `localhost:5432`, scan the VPC, and exfiltrate to an attacker C2. Docker provides zero isolation here — `--network=host` is the explicit opt-out from network namespacing.
>
> 2. **Single shared Bearer token** = no per-user revocation, no audit attribution, no rotation path. First leak (CI logs, screenshot, curl history) compromises every tenant simultaneously and silently.
>
> 3. **Container reused across requests** = script N+1 inherits filesystem state, env vars, cached secrets, and any backdoor (cron, LD_PRELOAD, modified `/usr/bin/*`) planted by script N. Cross-tenant contamination is the default, not the edge case.
>
> [...] **Verdict: BLOCK.**

Same verdict. Distinct, falsifiably-different reasoning. Neither peer saw the other's reply.

The full replies live at `~/.neil/logs/persona_em_reply.txt` and `~/.neil/logs/persona_cso_reply.txt`. The role-personas live at `personas/eng-mgr.md` and `personas/cso.md` (~83KB and ~65KB respectively, generated by the `bin/build-persona` translation pipeline from upstream gstack skills with `--upstream-sha` pinned).

## Demo 2 — Neil reviews its own commits, every 30 minutes, no human in the loop

A cron at `20,50 * * * *` runs `bin/neil-auto-review`. Each tick: read `state/neil-auto-review.json` for the last reviewed SHA, run `git log` since, hand the commit range to a peer Neil running the `/review` skill, and append every `[P0]`/`[P1]`/`[P2]` finding into `intentions.json` as fulfillment-contract `INTEND:` lines.

Real `logs/auto_review.log` from the past four hours:

```
[2026-04-27T18:19:56Z] first run — reviewing only HEAD (3f45456) against parent (5f1eda7)
[2026-04-27T18:21:24Z] review ok — 1 commits reviewed, 9 [P*] findings appended as INTENDs
[2026-04-27T18:50:02Z] reviewing 3f45456..c6ee255
[2026-04-27T18:52:59Z] review ok — 1 commits reviewed, 14 [P*] findings appended as INTENDs
[2026-04-27T19:20:01Z] reviewing c6ee255..a1c4197
[2026-04-27T19:21:07Z] review failed rc=1 (head=a1c4197); leaving last_review_sha unchanged
[2026-04-27T19:21:07Z] stderr: [handler] validate_params service=gstack action=review unknown
[2026-04-27T19:50:01Z] reviewing c6ee255..a1c4197
[2026-04-27T19:53:06Z] review ok — 2 commits reviewed, 12 [P*] findings appended as INTENDs
[2026-04-27T20:20:01Z] reviewing a1c4197..a26832c
[2026-04-27T20:21:28Z] review ok — 1 commits reviewed, 6 [P*] findings appended as INTENDs
```

Notice the recovery: at 19:20 the review fails (a parameter validation regression on the bridge), and the script correctly **does not advance** `last_review_sha`. At the next tick, it retries against the same range and succeeds with 12 findings. No commit slips past the review gate.

Sample `[P0]` finding from one of those runs (now an active intention Neil owes itself to resolve):

> [P0] gstack handler accepts `cwd=` parameter without realpath-allowlisting against `/home/seal` — a peer can request `cwd=/etc` and the spawned skill will operate there. Fix: realpath + prefix check in `handler.sh:validate_params`.

Neil filed this against itself. A subsequent commit closed it. The next review pass will check the closure.

## Demo 3 — The substrate is uniform; only configuration varies

The same C autoPrompter daemon runs in main Neil, in every peer Neil, and in every ephemeral temp Neil. The difference is which `essence/*.md` files load and which `services/registry/*.md` services are reachable. A peer Neil's `essence/persona.md` is loaded **last** in the file sort so its role voice survives drift even after lessons.md grows ([autoprompt.c `load_essence`](../tools/autoPrompter/src/autoprompt.c)).

```
~/.neil/                          ~/lvl2-em/.neil/                ~/lvl2-cso/.neil/
  essence/                          essence/                        essence/
    soul.md                          soul.md                          soul.md
    actions.md                       actions.md                       actions.md
    lessons.md                       lessons.md (peer-local)          lessons.md (peer-local)
    persona.md ← default             persona.md ← eng-mgr (83KB)      persona.md ← cso (65KB)
  tools/autoPrompter/                tools/autoPrompter/              tools/autoPrompter/
    src/autoprompt.c    ════════════ identical binary ═══════════════ src/autoprompt.c
```

Linux `MAX_ARG_STRLEN` is 128KB per argv element. When essence + an 80KB role persona collides with that ceiling, autoPrompter writes the system prompt to a tempfile and passes `--system-prompt-file <path>` to the bundled Claude CLI. The agent's SDK loader maps this to `SystemPromptFile{type:"file", path:...}` so the CLI reads from disk instead of being given an oversized argv. ([commit](https://github.com/arian-shamaei/neil/commit/8368ff3))

## Falsifiability

Every claim in this page corresponds to a file or log on the live host that you can read yourself if you spin up Neil:

| Claim | Evidence file |
|-------|---------------|
| Two peers reach BLOCK with distinct reasoning | `logs/persona_em_reply.txt`, `logs/persona_cso_reply.txt` |
| Persona library survives gates 1+2+3 (4 of 7 sub-gates green) | `logs/persona_falsification.log` |
| Auto-review runs every 30 min and recovers from bridge errors | `logs/auto_review.log` |
| Each peer produces SHA-256-distinct artifacts | `state/parallel_benchmark.jsonl` |
| Build-persona pins to a single gstack SHA | `bin/build-persona` (`UPSTREAM_SHA=...`) |
| The gstack bridge has been hardened across 3 review rounds | git log: `5c3473a` → `ab825ac` → `3c60c17` |

The verify scripts that produce these claims live in `self/verify/`. Each is a bash script that exits non-zero on regression. Cron runs them. Failures land in `self/failures.json`. Fixes land in commits.

## Try it

```bash
git clone https://github.com/arian-shamaei/neil.git ~/.neil
cd ~/.neil && ./install.sh
sudo systemctl start autoprompt
neil-blueprint    # Alt+8 → cluster panel

# Inside the TUI, type:
spawn a peer named alpha with persona=eng-mgr and ask it to review the webhook eval proposal in personas/eng-mgr.md
```

(LXD snap required for `spawn_vm`. Self-heals on first call.)
